📚 Explicación de Lifecycle Hooks

🏗️ constructor()

Cuándo: Cuando Angular crea el componente (PRIMERO)

Uso: Inyección de dependencias, inicialización básica

⚠️ Evitar: Lógica compleja, llamadas HTTP

constructor(private service: MyService) {{ '{' }}
  console.log('Componente creado');
{{ '}' }}

🔄 ngOnChanges()

Cuándo: Cuando cambian los valores de @Input

Uso: Reaccionar a cambios en propiedades de entrada

✅ Puede ejecutarse: Múltiples veces

ngOnChanges(changes: SimpleChanges) {{ '{' }}
  console.log('Input cambió:', changes);
{{ '}' }}

✅ ngOnInit()

Cuándo: Una vez, después del primer ngOnChanges

Uso: Inicializar datos, llamadas HTTP, suscripciones

✅ Lugar ideal para: Cargar datos del servidor

ngOnInit() {{ '{' }}
  this.loadData(); // Cargar datos
  this.initForm();  // Inicializar formulario
{{ '}' }}

🔍 ngDoCheck()

Cuándo: En cada ciclo de detección de cambios

Uso: Detección de cambios personalizada

⚠️ Cuidado: Se ejecuta MUY frecuentemente

ngDoCheck() {{ '{' }}
  // Detectar cambios manualmente
  // ⚠️ Evitar operaciones costosas
{{ '}' }}

📦 ngAfterContentInit()

Cuándo: Después de proyectar contenido externo

Uso: Trabajar con <ng-content>

✅ Se ejecuta: Una vez

ngAfterContentInit() {{ '{' }}
  // Contenido proyectado disponible
  console.log(this.contentChild);
{{ '}' }}

👁️ ngAfterViewInit()

Cuándo: Después de inicializar la vista del componente

Uso: Acceder a @ViewChild, @ViewChildren

✅ ViewChild disponible: Aquí es seguro usarlo

ngAfterViewInit() {{ '{' }}
  // ViewChild ya está disponible
  this.myInput.nativeElement.focus();
{{ '}' }}

💀 ngOnDestroy()

Cuándo: Justo antes de destruir el componente

Uso: Limpiar suscripciones, timers, listeners

⚠️ Importante: Evitar memory leaks

ngOnDestroy() {{ '{' }}
  clearInterval(this.timer);
  this.subscription.unsubscribe();
{{ '}' }}

✔️ Otros Hooks

ngAfterContentChecked: Después de verificar contenido

ngAfterViewChecked: Después de verificar vista

⚠️ Nota: Se ejecutan muy frecuentemente

// Se ejecutan en cada ciclo
// de detección de cambios
ngAfterViewChecked() {{ '{' }}
  // ...
{{ '}' }}

💡 Mejores Prácticas

✅ SÍ hacer

  • Usar ngOnInit() para cargar datos
  • Limpiar en ngOnDestroy()
  • Acceder a ViewChild en ngAfterViewInit()
  • Mantener constructor simple (solo DI)
  • Usar ngOnChanges() para reaccionar a @Input

❌ NO hacer

  • Lógica compleja en constructor
  • Olvidar limpiar suscripciones
  • Operaciones costosas en ngDoCheck()
  • Acceder a ViewChild en constructor
  • Llamadas HTTP en constructor
